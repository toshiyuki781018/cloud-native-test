# Kubernetes の全体像と主要コンポーネント  ― 分散した実行単位を成立させ続けるという話

第1章では、コンテナによって実行単位が環境から切り離された  
という変化を見てきました。

この章では、その次に現れる問題、分散した単位を、どう成立させ続けるのか  
という問いを扱います。

---

## 1. コンテナが増えた世界で何が問題になるのか

コンテナによって、アプリケーションは環境から切り離され、  
同じ形で再現でき、必要に応じて作って壊せるようになりました。

しかしこの変化は、同時に新しい問題を生みます。

- 数が増える  
- 動く場所が固定されない  
- 状態が常に変わる  

このとき起きるのは、単体は正しくても、全体は崩れるという状態です。

ここで問われるのは、  
**「どう作るか」** ではなく **「どう成立させ続けるか」** です。

---

## 2. Kubernetes は何を解決しようとしているのか

Kubernetesは、コンテナを動かすツールではありません。  
Kubernetesが扱うのは、**「あるべき状態」** です。

たとえば、

- いくつ動いていてほしいか  
- どこで動いていてほしいか  
- 壊れたときどう戻るべきか  

といった状態を受け取り、それが成立するように動き続けます。  
ここで重要なのは、Kubernetesは「中身」を扱わないという点です。

- アプリケーションのロジック  
- 実装の詳細  

これらは責任範囲ではありません。  
Kubernetesはあくまで、状態を成立させる仕組みです。

---

## 3. なぜ「操作しない」という設計になるのか

従来の運用は「操作」が中心でした。

- 起動する  
- 停止する  
- 再起動する  

しかし分散環境では、この前提が崩れます。

- 状態が変わり続ける  
- 操作してもすぐにズレる  
- 人が追いかけられない  

そこでKubernetesは、操作ではなく「状態の宣言」という設計を採ります。

**「こうしてほしい」** ではなく  
**「こういう状態であってほしい」** を定義する。

そしてKubernetesは、**現在の状態**と**理想の状態**を比較し続け、  
その差分を埋めます。

ここで重要なのは、人が賢くなるのではなく、仕組みが賢くなるという前提です。

---

## 4. Kubernetes の全体像  ― Control Plane と Node

この仕組みは、明確な役割分担によって成立しています。

- Control-Plane：状態を理解し、調整する  
- Node　　　　：実際にコンテナを動かす  

ここで重要なのは、命令ではなく「状態の共有」で動くという点です。

- 状態はAPIで共有される  
- 各コンポーネントがそれを見て動く  

この構造により、一部が壊れても、他がそれを補い、全体は維持される  
という性質が生まれます。

---

## 5. 主要コンポーネントは何をしているのか

Kubernetesのコンポーネントは、「機能」ではなく「責務」で分かれています。

- API-Server：状態の入口  
- Scheduler：配置の決定  
- Controller：状態の修正  
- kubelet  ：Node上での実行  

それぞれが一つの責務に集中しています。  
これは、壊れ続ける前提で設計されているためです。

一部が壊れても、他が補完し、全体は成立し続ける。  
この構造が、分散環境を支えています。

---

## 6. Kubernetes は OS ではない

Kubernetesは「分散OS」と言われることがあります。  
しかしこの理解は不正確です。

OSは、

- プロセスを動かす  
- メモリを管理する  
- ハードウェアを扱う  

という役割を持ちます。

一方Kubernetesは、状態を維持する制御を行う仕組みです。

- 実際の実行 → OS / コンテナランタイム  
- 状態の維持 → Kubernetes  

つまりKubernetesは、「OSの上に乗る制御層」です。

この理解がないと、Kubernetesがすべて解決するからOSを知らなくてよい  
という誤解につながります。

---

## 7. この章のまとめ  ― 次に見える問題

Kubernetesによって、単位は自動で配置され、
状態は監視され、ズレは修正される  という構造が成立しました。

つまり、制御は人から仕組みへ移されたと言えます。

しかしここで、新しい問いが生まれます。  
その単位は「どこで動かすのか」です。

- クラウドか  
- オンプレミスか  
- どこまでを自分で持つのか  

この選択は、

- コスト  
- 可用性  
- 責任範囲  

に直結します。

つまりここで初めて、「環境」が設計の一部として現れるのです。
