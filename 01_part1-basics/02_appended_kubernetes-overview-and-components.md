# Kubernetes の全体像と主要コンポーネント  ～分散した実行単位を成立させ続けるという話～

第1章では、コンテナによって実行単位が環境から切り離された  
という変化を見てきました。

この章では、その次に現れる問題、分散した単位を、どう成立させ続けるのか  
という問いを扱います。

---

## 1. コンテナが増えた世界で何が問題になるのか

コンテナによって、アプリケーションは環境から切り離され、  
同じ形で再現でき、必要に応じて作って壊せるようになりました。

これは大きな前進でした。

しかしこの変化は、同時に新しい問題を生みます。

- 実行単位が増える  
- 実行場所が固定されない  
- いつ壊れるか分からない   

このとき起きるのは、単体では正しく動くアプリケーションも、  
数が増え、場所が分散すると、全体としては簡単に破綻します。

ここで問題になるのは、**「どう作るか」** ではなく   
**「どう成立させ続けるか」** という点です。

---

## 2. Kubernetes は何を解決しようとしているのか

Kubernetesは、コンテナを動かすツールではありません。

Kubernetes が向き合っているのは、分散した実行単位を  
人の手に頼らず成立させ続けることです。

そのため、Kubernetesが扱うのは操作ではなく、**あるべき状態（Desired State）** です。

例えば、

- アプリケーションはいくつ動いていてほしいのか  
- どこで動いていてほしいのか  
- 壊れたときに、どういう状態で戻ってほしいのか  

これら状態の定義を受け取り、**現在の状態と理想の状態の差分を埋め続ける**のが  
Kubernetes の役割です。

ここで重要なのは、Kubernetesはアプリケーションの中身には関与しないという点です。  

- アプリケーションの実装  
- ビジネスロジック

これらはKubernetesの責任範囲ではありません。  
Kubernetesはあくまで、「成立の条件」だけを扱う存在です。

ここを切り分けて考えることが、Kubernetesを理解する最初のポイントです。

---

## 3. なぜ人が操作しないという設計思想になるのか

Kubernetes を初めて触った人が戸惑う点の一つに、  
**なぜ、コンテナを1つずつ直接操作しないのか?** という疑問があります。

従来の運用では

- 起動する  
- 停止する  
- 再起動する

こういった、人が直接手を動かす操作が中心でした。  
しかし分散環境では、この前提が崩れます。

なぜなら、常に状態が変わり続け、操作してもすぐにズレが発生し、  
人が追いかけられないほどの変化が発生するからです  

ここでKubernetesは、操作ではなく**状態を宣言する**という設計を採ります。

**「こう操作する」** ではなく  
**「こういう状態であってほしい」** を定義する。

そしてKubernetesは、**状態のズレを検知し、常に修正し続ける**ことで全体を成立させます。  
つまりここで、制御は人の判断から仕組みへと移されるという変化が起きます。

ここで重要なのは、人が賢くなるのではなく、仕組みが賢くなるという前提です。

---

## 4. Kubernetes の全体像  ― Control Plane と Node の役割

Kubernetesの役割は**現在の状態と理想の状態の差分を埋め続ける**仕組みです。  
この仕組みを実現するために、Kubernetes は役割を分離しています。

- Control Plane
- Node

Control Plane は、あるべき状態を理解し、維持するための判断を行う役割を持ちます。  
一方でNodeは実際にコンテナを動かす実行の場です。  

ここで重要なのは、ControlPlaneがNodeに対して、命令しているわけではないという点です。

Kubernetesの設計では状態はAPIで共有され、各コンポーネントがそれを見て自律的に動く  
という前提が置かれています。
   
この構造により、一部が壊れても、他がそれを補い、全体は維持される  
という性質が生まれます。

---

## 5. 主要コンポーネントは何をしているのか

Kubernetes には複数の主要コンポーネントが存在してます。  
ここで見るべきはこれらの主要コンポーネントがなぜこの分担になっているのかです。

- API-Server：状態の窓口  
- Scheduler：実行場所の決定  
- Controller：状態の修正  
- kubelet  ：Node上での実行

これらの主要コンポーネントは、「機能」ではなく「責務」で分かれており、  
それぞれが一つの責務に集中しています。

この分離によって、一部が壊れてもクラスターが停止せず、  
影響が限定されるという構造が保たれます。

これは、壊れ続ける前提で設計されているためです。

一部が壊れても、他が補完し、全体は成立し続ける。  
この構造が、分散環境である Kubernetesを支えています。

---

## 6. Kubernetes は OS ではない

Kubernetesは「分散OS」と言われることがあります。  
この表現は直感的にわかりやすいですが、本質を誤解させる可能性があります。

まず、OSが担っている役割を整理すると、

- プロセスの実行と制御
- メモリやネットワークの直接的な扱い
- ハードウェア資源の管理

いった「実行そのもの」を担います。

一方で Kubernetesは、状態を定義し、維持することに特化しています。  

つまり、Kubernetesは

- プロセスを起動する存在ではない  
- サーバーを直接管理する存在でもない  
- ハードウェア資源を扱う存在でもない  

ここで重要なのは、Kubernetes は「上に乗る制御の仕組み」であるという点です。

この構造を理解していないと、Kubernetesがすべてを解決してくれると解釈し、  
OSやネットワークを知らなくてもよいという誤解につながります。

実際には、Kubernetes は**責任を肩代わりしている部分**と  
**前提として残っている部分**を明確に分けています。

この違いは、次章以降で扱う実行環境や、  
その下にあるOS・ネットワークの話と接続することでよりはっきり見えてきます。

---

## 7. この章のまとめ  ― 次に見える問題

ここまでで、実行単位を成立させ続ける仕組みとして、  
Kubernetes を捉えることができました。

Kubernetesによって、単位は自動で配置され、  
状態は監視され、ズレは修正されるという構造が成立することになります。

つまり、制御は人から仕組みへ移される構造が成立します。

しかしここで、新しい問いが生まれます。  
その単位はどこで動かすのか？

コンテナは場所から切り離され、  
Kubernetesはその配置を制御できるようになりました。

だからこそ、

- どの環境で動かすのか
- どこまでを自分たちで制御するのか
- どこを仕組みに任せるのか
- クラウドか  
- オンプレミスか  
- どこまでを自分で持つのか  

という判断が、避けて通れなくなります。

この選択は、

- コスト  
- 可用性  
- 運用の責任範囲 

にシステム全体の前提に直結することになります。

つまりここで初めて、「どこで動かすか」という問題が、  
単なる環境選択ではなく、設計の一部として現れます。
