# 第3部：アプリケーションを届け・つなぐ技術

## この部の位置づけ

第2部では、

- 壊れる  
- 戻る  
- 任せる  

という挙動を体験し、クラウドネイティブの前提を理解してきました。

第3部では、その前提の上で、どのように設計するのかを扱います。  
ここからは、操作や理解ではなく、**判断の領域**に入ります。

---

## マイクロサービスはなぜ生まれたのか

マイクロサービスは、単なる技術の選択ではありません。

システムの規模が大きくなり、チームが増え、変更頻度が上がり、影響範囲が広がると、  
単一のアプリケーションでは管理が難しくなります。

マイクロサービスは、組織と変更速度のスケールに対応するための構造として生まれました。

---

## この部で扱うこと

マイクロサービス環境では、  
システムの中に二つの流れが生まれます。

### ① 変更という物流（変更の流れ）

- アプリケーションをどう更新するか  
- 変更をどう安全に届けるか  

### ② 通信という物流（通信の流れ）

- 分割されたサービスがどうつながるか  
- 通信の振る舞いをどこで制御するか  

この第3部では、アプリケーションを「届ける」「つなぐ」ための構造を整理します。

---

## この部の構成

この部では、次の三つの観点から整理します。

### 1. アプリケーションの構成

- どの単位で分割するのか  
- どのように配置するのか  

ここでは、

- UI / API / DB  
- Kubernetesリソース（Pod / Service / Ingress）  

を通して、クラウドネイティブアプリケーションの構造を理解します。


### 2. 変更を届ける仕組み  — 継続的デリバリーとGitOps —

変更が頻繁に発生する環境では、

- 変更をどう管理するか  
- 環境へどう反映するか  

これらがが重要になります。  
ここでは、変更という物流をどう設計するかを扱います。

### 3. サービス間通信制御  — Istio / Linkerd —

サービスが分割されると、通信はネットワーク越しになります。

ここでは、

- 失敗をどう扱うか  
- 通信をどこで制御するか  

という通信という物流をどう設計するかを扱います。

---

## この部で重要な視点

この部で扱うのは、ツールの使い方や技術の選び方ではありません。  
重要なのは、その複雑さをどこで引き受けるのかという設計です。

クラウドネイティブでは、アプリケーションを分割することで  
変更や通信という複雑さが生まれます。

それを、アプリケーションで持つのか、基盤に任せるのか、外部に委ねるのか。  
どこで責任を持つのかが、設計になります。

---

## この部を終えると

この部を終えると、次のことが判断できるようになります。

- なぜマイクロサービスにするのか  
- どこまで分割すべきか  
- 変更をどう扱うべきか  
- 通信の責任をどこに持たせるべきか  

つまり、技術を選ぶ前に、構造を選べる状態になります。
